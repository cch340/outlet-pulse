# Supabase Backend & Database — Design

**Date:** 2026-06-29
**Status:** Approved (pending spec review)

## Goal

Replace the in-memory data store in the Staff Monitor web app with a real,
persistent backend using **Supabase (free tier)**, and add full CRUD UI for the
core entities. The app currently holds all data in `seed.ts` loaded into a React
context `useState` store; after this work, data lives in Postgres and survives
reloads.

## Scope

In scope:
- Supabase Postgres schema for all domain entities.
- React app talks **directly** to Supabase via `@supabase/supabase-js` (no
  separate backend server).
- **TanStack Query** for all server state (caching, loading/error, refetch,
  optimistic updates).
- Full **CRUD UI** for brands, outlets, and staff (add / edit / delete).
- Existing write flows (staff transfer, create follow-up, toggle task, mark
  done, toggle status) migrated to Supabase mutations.

Out of scope (later projects):
- Authentication and per-user data scoping (RLS will be permissive for now).
- Realtime subscriptions.
- A standalone backend API server.

## Decisions (locked)

- **No auth yet.** Single permissive access; RLS disabled/permissive.
- **Direct-to-Supabase**, no backend server.
- **Start with an empty database** (no seed migration).
- **Full CRUD UI** for brands/outlets/staff (since none exists today).
- **TanStack Query** for data fetching/caching.
- **UUID primary keys** generated by Postgres (replacing string ids like `b1`).
- `staff_history` and `follow_up_tasks` are **child tables**, not JSON columns.

## Architecture

```
React (Vite) ──@supabase/supabase-js──► Supabase Postgres (REST + Postgres)
     │
     └── TanStack Query (cache, loading/error, optimistic updates)
```

- `src/lib/supabase.ts` — single Supabase client, configured from
  `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` (in `.env.local`, gitignored).
- The existing context store (`src/data/store.tsx`) **shrinks to client-only UI
  state**: `activeScreen`, `isMobile`, `period`, `q`, `selectedBrandId`,
  `selectedOutletId`, `staffBrandFilter`, `fuFilter`, overlay flags, and theme
  (`accent`, `themeMode`, `density`). These never touch the DB. Theme prefs may
  optionally persist to `localStorage`.
- Domain collections (`brands`, `outlets`, `stores`, `staff`, `followups`) move
  out of the store and become TanStack Query hooks.

## Database Schema

All tables use `uuid` primary keys with `default gen_random_uuid()`.

| Table | Columns | Notes |
|---|---|---|
| `brands` | id, name, color, category | |
| `outlets` | id, name, location | |
| `stores` | brand_id, outlet_id | join; PK = (brand_id, outlet_id); FKs to brands/outlets |
| `staff` | id, name, brand_id, outlet_id, role, joined (date) | FKs to brands/outlets |
| `staff_history` | id, staff_id, brand_id, outlet_id, from_label, to_label (nullable), reason (nullable) | append-only transfer log; FK to staff |
| `follow_ups` | id, date, staff_id (nullable), brand_id, outlet_id, status ('done'\|'pending') | FKs to staff/brands/outlets |
| `follow_up_tasks` | id, follow_up_id, label, done (bool), sort (int) | child rows; FK to follow_ups |

Schema rationale:
- `staff_history` as its own table: it is an append-only audit log, cleaner to
  insert into and query than a JSON array on `staff`.
- `follow_up_tasks` as a child table: the app toggles individual tasks, so a
  targeted single-row update beats rewriting a JSON blob.
- FKs use `on delete restrict` for brand/outlet references from staff and stores
  (so deletes are guarded — see CRUD UI), and `on delete cascade` for
  `staff_history` and `follow_up_tasks` (children die with their parent).

Delivered as a single SQL migration file the user runs in the Supabase SQL
editor. RLS left disabled (or a permissive `allow all` policy) with an inline
comment marking where per-user policies attach once auth lands.

## Data Layer

`src/data/queries/` — one module per entity:

- `useBrands.ts`: `useBrands()`, `useCreateBrand()`, `useUpdateBrand()`,
  `useDeleteBrand()`.
- `useOutlets.ts`, `useStaff.ts`, `useStores.ts`, `useFollowUps.ts`: same shape.
- Mutations invalidate the relevant query keys on success.

Existing write flows become mutations:
- **Transfer**: updates the `staff` row's brand/outlet **and** inserts a
  `staff_history` row (the previous open history entry gets its `to_label`
  stamped). Done as a single logical operation.
- **Create follow-up**: inserts a `follow_ups` row plus its selected
  `follow_up_tasks` rows.
- **Toggle task / mark done / toggle status**: targeted updates on
  `follow_up_tasks` / `follow_ups`.

Derived selectors in `src/data/derived.ts` are reworked to consume query data
instead of the store collections.

## CRUD UI

Follows existing screen + modal patterns (`TransferModal.tsx`,
`ScheduleModal.tsx`):

- **Brands**, **Outlets**, **Staff** screens each gain an **Add** button and
  per-row **Edit** / **Delete** actions.
- One small reusable form modal per entity (brand form, outlet form, staff form).
- Delete guards: deleting a brand or outlet still referenced by `stores` or
  `staff` is blocked with a clear message (enforced by FK `restrict` + a
  friendly pre-check in the UI).

## Phasing

Each phase leaves the app runnable.

1. **Schema** — SQL migration file; user creates the Supabase project and runs
   the migration. Document the env-var setup.
2. **Data layer** — supabase client, TanStack Query provider, query/mutation
   hooks; rewire existing read screens and existing write flows. App works
   end-to-end against an empty DB.
3. **CRUD UI** — add/edit/delete for brands, outlets, staff.

## What the user must do (outside the code)

1. Create a free Supabase project at supabase.com.
2. Run the provided SQL migration in the SQL editor.
3. Copy the project URL + anon key into `.env.local`.
4. Note: free-tier projects pause after ~7 days of inactivity (one click to
   resume); fine for dev/demo.

## Risks / Notes

- No auth means the anon key has broad access; acceptable for dev/demo, must be
  revisited before any real launch. RLS attachment points are flagged in the
  migration.
- Async data introduces loading/error states on every screen — TanStack Query
  standardizes this, but screens need loading/empty/error handling added.
